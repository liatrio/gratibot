# Task 3.1 Proofs — `files.sharedPublicURL` spike (SPIKE_FALLBACK)

## Task Summary

Before writing any Unit 3 code, the spec required a live probe against the
dev Slack workspace to confirm whether a **bot token** can successfully call
`files.sharedPublicURL` on an uploaded file. The outcome is binary and
gates the scope of Unit 3 implementation:

- `SPIKE_OK` → ship the full in-modal `file_input` upload flow.
- `SPIKE_FALLBACK` → drop in-modal upload from v1; Unit 2's plain-text
  `imageURL` field remains the admin form's image input; defer file upload
  to a follow-up spec.

## Decision

**SPIKE_FALLBACK** — the bot token is rejected by `files.sharedPublicURL`
with the documented `not_allowed_token_type` error, matching the exact
outcome the spec anticipates in §Technical Considerations → Image Handling
and in [slackapi/node-slack-sdk#1000](https://github.com/slackapi/node-slack-sdk/issues/1000).

## What This Task Proves

- The call's pre-requisite scopes (`files:read`, `files:write`) are
  necessary but not sufficient: even with both scopes granted on the bot
  token, Slack refuses the request due to token-type restrictions.
- The failure mode observed is the **exact** error the spec flagged as
  the trigger for the fallback plan — no workspace policy, no network
  error, no manifest drift. This is the documented behavior, not an
  environmental oddity.
- The fallback path specified by the spec (revert to plain-text
  `imageURL`, do not add scopes to the committed manifest, defer upload
  work) is the correct path forward.

## Evidence Summary

- **Probe 1** (before scopes) confirmed the bot is missing `files:read`
  (expected). This is a prerequisite check, not the spike result.
- **Probe 2** (after temporarily adding `files:read` + `files:write` to
  the dev Slack app's Bot Token Scopes and reinstalling) called
  `files.sharedPublicURL({ file: F0AU1RHDDCM })` and received
  `ok: false, error: "not_allowed_token_type"`. This is the spike result
  and triggers `SPIKE_FALLBACK`.

## Artifact: Probe 1 — baseline `files.list` failure

**What it proves:** The dev bot token's scope set before the spike did not
include `files:read`, so no file operations are possible without the
scope change. This rules out "we already had the scopes and it still
failed for some other reason."

**Why it matters:** Establishes the starting state cleanly so Probe 2's
result is attributable to token-type, not to missing scopes.

**How it was invoked:** A local Node script loaded
`BOT_USER_OAUTH_ACCESS_TOKEN` from `.env` and called
`new WebClient(token).files.list({ count: 10 })`. A minimal
reproducibility recipe is included at the end of this document
(Appendix: Reproducibility).

**Result summary:** `files.list` responded `ok: false, error:
"missing_scope", needed: "files:read"` with the full existing scope list
enumerated — confirming the bot token lacked `files:read` (and, by
extension, `files:write`) before the spike modification.

```json
{
  "ok": false,
  "error": "missing_scope",
  "needed": "files:read",
  "provided": "app_mentions:read,channels:history,channels:read,channels:join,chat:write,groups:history,groups:read,im:history,im:read,im:write,mpim:history,mpim:read,mpim:write,reactions:read,reactions:write,usergroups:read,users:read",
  "response_metadata": {
    "scopes": [
      "app_mentions:read",
      "channels:history",
      "channels:read",
      "channels:join",
      "chat:write",
      "groups:history",
      "groups:read",
      "im:history",
      "im:read",
      "im:write",
      "mpim:history",
      "mpim:read",
      "mpim:write",
      "reactions:read",
      "reactions:write",
      "usergroups:read",
      "users:read"
    ],
    "acceptedScopes": ["files:read"]
  }
}
```

## Artifact: Probe 2 — `files.sharedPublicURL` with scopes granted

**What it proves:** Even with `files:read` and `files:write` added to the
bot token and the app reinstalled, `files.sharedPublicURL` refuses the
bot token and returns `not_allowed_token_type`. This is the exact
fallback-trigger condition the spec identified.

**Why it matters:** This is the single load-bearing fact for the
`SPIKE_FALLBACK` decision. Without it, we would be deferring a feature
on speculation instead of a recorded observation.

**Setup:** `files:read` and `files:write` were added to **Bot Token
Scopes** in the dev Slack app's OAuth & Permissions panel; the app was
reinstalled to the workspace; a test PNG was uploaded into a DM with
dev Gratibot to produce file id `F0AU1RHDDCM`.

**How it was invoked:** A local Node script loaded the refreshed
`BOT_USER_OAUTH_ACCESS_TOKEN` from `.env` and called
`new WebClient(token).files.sharedPublicURL({ file: "F0AU1RHDDCM" })`.
See the reproducibility recipe in the appendix below.

**Result summary:** Response was `ok: false, error:
"not_allowed_token_type"`. No `files:write` scope-level error (we had
the scope); no network error; no workspace-policy error. The method
itself refused the token type, matching the documented behavior for
bot tokens.

```json
{
  "message": "An API error occurred: not_allowed_token_type",
  "data": {
    "ok": false,
    "error": "not_allowed_token_type",
    "response_metadata": {}
  }
}
```

## Scope of the Fallback

Every Unit 3 Functional Requirement is deferred to a follow-up spec.

| Unit 3 FR (from spec §Unit 3) | Disposition |
| --- | --- |
| Replace `imageURL` plain-text input with `file_input` in Add/Edit forms | **Deferred.** Unit 2's plain-text `imageURL` input remains the admin form's image field. |
| Convert uploaded file reference to a public URL via `files.sharedPublicURL` and persist | **Deferred.** Not feasible with a bot token. |
| Return `response_action: "errors"` on upload failure | **Deferred.** No upload path in v1. |
| Add `files:read` + `files:write` to the bot scope manifest | **Not applied.** The temporary grant used in Probe 2 lives only in the dev Slack app's OAuth panel and must be removed there (see Follow-ups below). `slack_app_manifest.yml` is unchanged in this commit. |
| Preserve existing Azure-Blob-hosted `imageURL` values from seeded rewards | **Already satisfied** by Unit 1's seed + Unit 2's `updateReward` not touching fields absent from the `$set` payload. No action required. |

## Follow-ups

1. **Revert dev-app scopes** — the temporary `files:read` / `files:write`
   grant on the dev Slack app should be removed from its OAuth &
   Permissions panel and the app reinstalled, since we are not shipping
   the scope addition. Production and nonprod bots were never modified.
2. **Follow-up spec** — a future spec can revisit file upload using one
   of the paths the spike did not rule out:
   - User-token flow (requires a human-owned OAuth grant, not a bot
     token).
   - Post-uploaded-file-to-a-workspace-channel-then-permalink workaround
     (explicitly out of scope for this spec per §Technical
     Considerations → Image Handling).
   - External object store (Azure Blob) with an admin-pasted URL,
     matching the current manual flow.

## Reviewer Conclusion

The live probe produced the specific error code the spec identified as
the fallback trigger. Proceeding with a `SPIKE_FALLBACK` docs-only
commit for Unit 3 is aligned with the spec's written resolution
approach. Unit 4 and Unit 5 are unaffected.

## Appendix: Reproducibility

To rerun the probe in the future (e.g., to check whether Slack has
changed the bot-token restriction on `files.sharedPublicURL`), paste
the snippet below at the repo root. No helper script is kept in the
repo — the dependency graph already ships `@slack/web-api` transitively
via `@slack/bolt`.

```bash
# 1. Make sure .env at repo root has BOT_USER_OAUTH_ACCESS_TOKEN=xoxb-...
# 2. Temporarily grant files:read + files:write on the dev Slack app's
#    Bot Token Scopes, then reinstall the app.
# 3. Upload a test image to a DM with dev Gratibot so it's accessible.
# 4. List files to find the uploaded file's ID:
node -e '
const { WebClient } = require("@slack/web-api");
const token = require("fs").readFileSync(".env","utf8")
  .match(/BOT_USER_OAUTH_ACCESS_TOKEN=(.*)/)[1].trim();
new WebClient(token).files.list({ count: 10 })
  .then(r => console.log(JSON.stringify(r, null, 2)))
  .catch(e => console.log(JSON.stringify({ message: e.message, data: e.data }, null, 2)));
'

# 5. Probe sharedPublicURL with that file ID (replace F0123ABCDE):
node -e '
const { WebClient } = require("@slack/web-api");
const token = require("fs").readFileSync(".env","utf8")
  .match(/BOT_USER_OAUTH_ACCESS_TOKEN=(.*)/)[1].trim();
new WebClient(token).files.sharedPublicURL({ file: "F0123ABCDE" })
  .then(r => console.log(JSON.stringify(r, null, 2)))
  .catch(e => console.log(JSON.stringify({ message: e.message, data: e.data }, null, 2)));
'

# 6. Remove the temporary scopes from the dev Slack app afterwards.
```
